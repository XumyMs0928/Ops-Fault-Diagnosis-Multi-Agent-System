 我构建了一个基于多 Agent 协作的运维故障诊断系统（Ops Fault Diagnosis Multi-Agent System）。它能在生产环境发生告警风暴
  时，自动完成从告警降噪、根因定位、修复建议到复盘报告的全链路诊断闭环，将传统依赖资深 SRE 经验的故障定位过程自动化。

  核心痛点： 线上故障响应严重依赖少数资深 SRE 的经验。一次典型的数据库故障可能在 5 分钟内产生 50+ 条关联告警，横跨 6
  个微服务，导致告警风暴。新人面对海量告警无从下手，平均故障定位时间（MTTI）长达 30-60
  分钟，且故障处理知识难以沉淀和传承。每个故障的复盘报告也往往因为时间压力而流于形式。

  核心逻辑流： 系统采用 4 个专业化 Agent 顺序协作的架构，通过共享上下文（SharedContext）进行状态传递，每个 Agent
  拥有独立的系统提示词和推理策略：

  1. 告警聚合 Agent：接收原始告警流，基于服务调用拓扑进行去重和关联归约。将 50+ 条原始告警降噪为 1-2 个高价值事件，避免
  告警风暴淹没关键信息。核心利用拓扑感知的关联算法——同一调用链上相邻服务在短时间窗口内产生的告警被视为同一事件。
  2. 根因定位 Agent：这是系统最核心的环节，实现了多步长链推理。它不会直接跳到结论，而是从表面症状出发，沿服务调用链逐层
  下钻：先是追踪调用链定位疑似根因服务，然后深入检查该服务的日志和指标异常，再交叉比对近期部署和配置变更，最终综合形成完
  整的证据链和根因分析。典型场景下需要 3-5 轮与 Claude
  的交互推理，每一步都有明确的"检查了什么→观察到了什么→意味着什么"的推理记录。
  3. 修复建议 Agent：基于根因分析结果，生成包含立即止血方案和长期预防方案的修复计划。针对已知故障模式（如连接池耗尽、OOM
   Kill、磁盘满等），能直接给出可执行的 kubectl 命令或 SQL 脚本，并标注风险等级和置信度。
  4. 复盘 Agent：在故障处理完成后，自动生成结构化的复盘报告，包含时间线重建、影响评估、改进项追踪（含优先级和责任人），
  以及可传承的经验教训，将隐性的 SRE 经验显性化。

  系统内置了 3 个典型生产故障场景的完整模拟数据（数据库连接池耗尽级联、内存泄漏 OOM
  Kill、磁盘满级联），包含合成的告警、日志、指标时序和变更记录，开箱即可运行完整的端到端诊断流程。基于 Anthropic Claude
  API 实现 Agent 推理，采用流式输出 + 事件总线的架构，CLI 界面可实时展示每个 Agent 的推理过程，让多 Agent
  协作和长链推理可见可解释。
